# 크몽 판매용 모바일 홍보페이지 빌더 작업 이어가기 요약

작성일: 2026-06-09

## 한 줄 요약

크몽에서 팔 상품은 “완성형 랜딩페이지 제작 대행”이 아니라, `사진·판매 품목·가격·이벤트를 입력하면 모바일 홍보페이지를 한눈에 보고 편집하고, 마음에 드는 양식 1개를 저렴하게 구매하는 웹 기반 템플릿/빌더 상품`이다.

## 크몽 판매 가능성 판단

크몽에서 판매 가능성은 있다. 다만 포지셔닝을 잘못 잡으면 흔한 랜딩페이지 제작 서비스와 비교되어 묻힐 수 있다.

피해야 할 포지션:

```text
랜딩페이지 제작해드립니다
홈페이지 만들어드립니다
상세페이지 제작해드립니다
```

잡아야 할 포지션:

```text
사진과 가격표만 넣으면 완성되는 인스타 프로필용 모바일 안내페이지를 만들어드립니다
```

또는:

```text
가격표·이달의 이벤트·예약 안내를 직접 수정할 수 있는 모바일 홍보페이지 양식을 세팅해드립니다
```

## 핵심 차별점

1. 완성품 대행이 아니라 저렴한 양식/빌더 기반 상품이다.
2. 고객은 사진, 판매 품목, 가격 정보, 이벤트를 입력한다.
3. 오른쪽에서 모바일 페이지 미리보기를 한눈에 확인한다.
4. 같은 데이터로 여러 양식을 바꿔볼 수 있다.
5. 양식 하나당 저렴하게 판매한다.
6. 맞춤 제작은 기본 상품이 아니라 업셀로 둔다.

## 사용자 의도 정리

사용자가 원하는 것은 다음과 같다.

```text
웹페이지로 중계 웹사이트를 만든다.
양식 하나당 돈을 받고 판다.
완성품을 주는 게 아니므로 가격은 저렴해야 한다.
사진, 판매 품목, 가격 정보, 이벤트 등을 입력받는다.
클로드 디자인처럼 한눈에 보고 편집한다.
```

## 가격 정책 초안

크몽에서는 처음부터 너무 복잡하게 팔지 말고 3단계로 파는 것이 좋다.

### 기본형

가격 후보:

```text
19,000원 ~ 29,000원
```

제공:

```text
양식 1개
고객 정보 반영
모바일 안내페이지 HTML 또는 링크 제공
수정 1회
```

### 세팅형

가격 후보:

```text
49,000원 ~ 79,000원
```

제공:

```text
사진/가격표/이벤트 정리
양식 1개 세팅
문구 약간 다듬기
수정 2회
```

### 패키지형

가격 후보:

```text
99,000원 ~ 149,000원
```

제공:

```text
같은 데이터로 양식 3개 제작/비교
최종 1개 선택
공유 링크 또는 HTML 제공
수정 2~3회
```

주의:

- “9,900원”은 자체 웹사이트에서 자동 판매할 때 적합하다.
- 크몽에서는 상담/응대/수정이 들어가므로 너무 낮게 잡으면 노동비가 안 나온다.
- 크몽 1차 상품은 19,000원 이상이 현실적이다.

## 크몽 상품 제목 후보

1. `사진과 가격표만 주시면 인스타 프로필용 모바일 안내페이지를 제작해드립니다`
2. `소상공인 가격표·이벤트 모바일 랜딩페이지를 저렴하게 만들어드립니다`
3. `네일샵/공방/필라테스용 모바일 안내페이지 양식을 세팅해드립니다`
4. `가격표와 이벤트가 보이는 모바일 홍보페이지를 제작해드립니다`
5. `인스타 링크용 모바일 메뉴판·예약 안내페이지를 만들어드립니다`

가장 추천:

```text
사진과 가격표만 주시면 인스타 프로필용 모바일 안내페이지를 제작해드립니다
```

```text
비싼 홈페이지가 부담스러운 사장님을 위한
인스타 프로필용 모바일 안내페이지입니다.

사진, 가격표, 이벤트, 예약 안내를 한 화면에 정리해드립니다.
네일샵, 공방, 필라테스, 피부관리, 에어컨청소 등
소상공인 홍보에 맞춘 모바일 전용 페이지입니다.
```

## 크몽에서 조심할 점

1. “직접 수정 가능”을 너무 강하게 쓰면 실제 수정 시스템을 요구받을 수 있다.
2. 현재 단계에서는 “수정 가능한 구조로 제작” 또는 “추후 수정이 쉬운 양식형 페이지” 정도가 안전하다.
3. “무제한 수정”은 절대 넣지 말 것.
4. 사진 보정/카피라이팅/브랜딩 전략은 기본가에 포함하지 말 것.
5. HTML 파일 제공인지, 링크 제공인지, 이미지 캡처 제공인지 명확히 해야 한다.
6. 고객이 직접 서버에 올릴 수 없는 경우가 많으므로 링크 발급/호스팅 옵션이 필요하다.

## 핵심 제품 관점 업데이트

사용자의 핵심 가설은 `AI가 로고 특징을 고르면 결과물을 뽑아주는 사이트는 이미 많지만, 클릭 몇 번으로 사진·가격·이벤트·매장 정보를 넣어 랜딩페이지/모바일 안내페이지를 자동 제작하고 파는 셀프 제작 사이트는 아직 드물다`는 것이다.

이유 판단:

- 로고/이미지 생성은 AI 결과물을 단품으로 팔기 쉽다.
- 반면 랜딩페이지는 아직 디자이너 외주 영역에 가깝다.
- 소상공인은 전문 디자이너에게 외주를 맡기기엔 부담스럽고, 직접 만들기엔 디자인/구성/배포가 어렵다.
- 따라서 기회는 “디자인 외주”가 아니라 “클릭 몇 번으로 자기 사진과 정보를 넣으면 자동으로 붙는 셀프 제작 사이트”에 있다.

제품의 본질:

```text
템플릿을 파는 사이트가 아니라,
소상공인이 자기 사진·상품·가격·이벤트·예약 정보를 넣으면
AI/자동화가 모바일 랜딩페이지에 맞게 배치해주고,
마음에 드는 양식을 골라 구매/다운로드/링크 발급하는 셀프 제작 플랫폼.
```

차별화 문장:

```text
로고처럼 AI가 뽑아주는 결과물을 사는 시대는 이미 왔다.
하지만 소상공인 랜딩페이지는 아직 외주의 영역에 머물러 있다.
우리는 그 과정을 클릭 몇 번짜리 셀프 제작으로 바꾼다.
```

MVP 방향 수정:

1. 단순 HTML 양식 판매보다 `셀프 제작 흐름`을 더 강하게 보여줘야 한다.
2. “양식 선택”보다 “내 정보 입력 → 자동 배치 → 결과 비교 → 구매” 플로우가 핵심이다.
3. 입력 패널은 엑셀 느낌보다 창업자/사장님이 쉽게 따라가는 단계형 질문 UI가 적합하다.
4. AI의 역할은 완전 자동 생성보다, 사용자가 넣은 정보와 사진을 보기 좋은 랜딩페이지 구성으로 자동 배치/문구 정리/양식 추천하는 쪽이 현실적이다.
5. 판매 문구는 “랜딩페이지 제작”이 아니라 “사장님이 직접 만드는 인스타 링크용 모바일 안내페이지 자동 제작기”에 가깝게 잡는다.

## 네이버 파워링크 판매 관점 업데이트

사용자는 이 상품을 크몽뿐 아니라 `네이버 파워링크로 직접 판매한다`는 관점으로 보는 것이 좋아 보인다고 판단했다.

전략적 의미:

- 크몽은 이미 제작 의뢰를 하려는 사람이 들어오는 마켓이라 경쟁자가 많고 가격 비교가 심하다.
- 네이버 파워링크는 “홈페이지 제작”, “랜딩페이지 제작”, “인스타 링크 페이지”, “가격표 디자인”, “네일샵 가격표”, “소상공인 홈페이지”처럼 문제를 검색하는 사람에게 직접 노출할 수 있다.
- 특히 `외주 맡기기엔 비싸고, 직접 만들긴 어려운` 소상공인을 자체 랜딩페이지로 받는 구조가 가능하다.
- 따라서 MVP는 크몽 상세페이지뿐 아니라 네이버 검색광고 유입용 판매 랜딩페이지로도 설계해야 한다.

파워링크용 포지셔닝:

```text
비싼 홈페이지 대신,
사진과 가격표만 넣으면 완성되는
인스타 프로필용 모바일 안내페이지 자동 제작기
```

파워링크 검색 의도 후보:

1. 홈페이지 제작 비용이 부담되는 소상공인.
2. 인스타 프로필 링크에 가격표/예약/이벤트를 정리하고 싶은 사장님.
3. 네일샵/공방/필라테스/피부관리/청소업처럼 모바일 안내페이지가 필요한 업종.
4. 랜딩페이지 외주를 알아보다가 더 저렴하고 빠른 대안을 찾는 사용자.

광고/랜딩 구조:

1. 검색광고 제목에서 `저렴한 모바일 안내페이지`, `인스타 프로필 링크`, `사진·가격표만 입력`을 강조한다.
2. 랜딩 첫 화면에서 바로 편집기 체험 또는 샘플 생성 버튼을 보여준다.
3. “외주가 아니라 셀프 제작이라 저렴하다”는 이유를 명확히 설명한다.
4. 무료 미리보기는 워터마크 포함, 구매 후 워터마크 제거/링크 발급/HTML 다운로드로 전환한다.
5. 업종별 키워드 광고를 별도 세트로 나눈다: 네일샵, 공방, 필라테스, 피부관리, 청소, 클래스 등.

실험 아이디어:

- 네이버 파워링크 소액 테스트 예산으로 키워드별 클릭률/문의율을 본다.
- 광고 랜딩페이지는 완성형 상품 설명보다 “지금 무료로 내 페이지 만들어보기” CTA를 앞에 둔다.
- 초기에는 결제 자동화보다 `무료 시안 생성 → 카톡 문의/결제` 흐름으로 전환율을 검증해도 된다.

## 업종별 양식 데이터 확보 관점

사용자는 이 서비스를 제대로 만들려면 `수많은 업종에 대한 수많은 양식 데이터`가 필요하다고 판단했다. 각 업종별로 최소 3~5개의 양식이 있어야 사용자가 “고를 수 있다”고 느끼고 실제 이용 가능성이 생긴다.

전략적 의미:

- 이 서비스의 해자는 단순 편집기 코드가 아니라 `업종별 랜딩페이지 양식 데이터베이스`다.
- 업종마다 고객이 보고 싶어 하는 정보, 사진 비중, CTA, 신뢰 요소, 가격표 구조, 이벤트 표현 방식이 다르다.
- 따라서 모든 업종에 하나의 범용 양식을 제공하면 약해지고, 업종별 3~5개 선택지를 제공해야 상품성이 생긴다.
- 처음부터 모든 업종을 커버하려고 하면 제작량이 너무 커지므로, 광고 테스트용 핵심 업종부터 순차적으로 양식 팩을 만드는 방식이 적합하다.

우선순위 업종 후보:

1. 네일샵: 가격표, 이달의 아트, 예약, 전후 사진, 이벤트가 명확하다.
2. 피부관리/에스테틱: 전후 사례, 패키지 가격, 첫 방문 이벤트, 신뢰 요소가 중요하다.
3. 필라테스/PT: 시간표, 체험권, 강사/공간 사진, 상담 CTA가 중요하다.
4. 공방/클래스: 클래스 일정, 작품 사진, 수강료, 예약/문의가 중요하다.
5. 청소/시공/방문 서비스: 전후 사진, 서비스 범위, 가격 기준, 상담 문의가 중요하다.

업종별 최소 양식 수 기준:

```text
업종 1개당 최소 3개, 가능하면 5개 양식
```

권장 양식 구성:

1. 기본 안내형: 가격표/서비스/예약 정보 중심.
2. 포스터 이벤트형: 할인/이벤트/체험권 중심.
3. 갤러리/룩북형: 사진 결과물 중심.
4. 예약/상담 전환형: CTA와 문의 흐름 중심.
5. 사례/후기형: 전후 사진, 리뷰, 신뢰 요소 중심.

제작 방식:

- 양식 HTML을 많이 만드는 것보다 먼저 `업종별 정보 구조 스키마`를 정의한다.
- 예: 네일샵은 `대표 사진, 이달의 아트, 가격표, 이벤트, 예약 안내`; 청소업은 `전후 사진, 공간 유형, 평수/범위, 가격 기준, 상담 CTA`.
- 같은 레이아웃 엔진을 쓰되 업종별 섹션 조합과 문구 톤, 샘플 데이터, 이미지 비율을 다르게 만든다.
- 데이터 축적이 경쟁력이라서, 제작 결과물은 `업종 x 양식 타입 x 샘플 데이터` 형태로 관리해야 한다.

MVP 다음 단계:

1. 네일샵 5종을 먼저 완성도 있게 만든다.
2. 그 다음 피부관리, 필라테스, 공방, 청소업 순서로 각 3종씩 확장한다.
3. 네이버 파워링크는 업종별 양식 팩이 준비된 업종부터 테스트한다.
4. 편집기 UI는 업종 선택 후 해당 업종 전용 질문/양식만 보여주는 구조로 바꾼다.

## 확정된 제품 플로우와 미리보기 원칙

사용자가 원하는 핵심 플로우는 다음과 같다.

```text
1. 업종 선택
2. 양식 선택
3. 양식별 필요한 정보 입력
4. AI에게 디테일한 수정 요청
5. 구매하면 워터마크 제거 및 링크 발급
```

기본 UI 원칙:

- 편집 중에는 `PC용 미리보기`와 `모바일용 미리보기`를 실시간으로 같이 보여준다.
- 모바일 안내페이지가 핵심이지만, 실제 구매자는 PC에서 제작할 가능성이 있으므로 PC/모바일 미리보기 동시 확인이 기본값이다.
- 사용자가 입력하거나 AI 수정 요청을 적용하면 양쪽 미리보기에 즉시 반영되어야 한다.

플로우별 역할:

1. 업종 선택
   - 네일샵, 피부관리, 필라테스, 공방, 청소 등 업종을 먼저 선택한다.
   - 업종 선택에 따라 필요한 정보 입력 항목과 추천 양식이 달라진다.

2. 양식 선택
   - 선택한 업종에 맞는 3~5개 양식을 보여준다.
   - 예: 기본 안내형, 포스터 이벤트형, 갤러리형, 예약 전환형, 사례/후기형.
   - 양식 선택 단계에서도 샘플 데이터가 들어간 PC/모바일 미리보기를 보여준다.

3. 양식별 필요한 정보 입력
   - 양식마다 필요한 입력 필드가 다르다.
   - 예: 예약형은 예약 가능 시간/문의 링크가 중요하고, 사례형은 전후 사진/후기가 중요하다.
   - 입력은 엑셀식 폼보다 사장님이 따라가기 쉬운 단계형 질문 UI가 적합하다.

4. AI에게 디테일한 수정 요청
   - 사용자는 “좀 더 고급스럽게”, “가격표를 위로”, “이벤트를 더 크게”, “20대 여성 취향으로”처럼 자연어로 수정 요청한다.
   - AI는 전체 재생성보다 현재 양식 안에서 문구, 섹션 순서, 강조도, 톤, CTA를 조정한다.
   - 수정 요청 후에도 PC/모바일 미리보기가 즉시 업데이트되어야 한다.

5. 구매 후 워터마크 제거 및 링크 발급
   - 무료 미리보기 단계에서는 워터마크를 표시한다.
   - 구매하면 워터마크 제거, 공유 링크 발급, 필요 시 HTML 다운로드를 제공한다.
   - 초기 검증 단계에서는 자동 결제 전에도 `워터마크 미리보기 → 카톡 문의/결제 → 링크 발급` 방식으로 시작 가능하다.

제품 정의 업데이트:

```text
업종을 고르고, 양식을 고르고, 필요한 정보를 넣은 뒤,
AI에게 세부 수정까지 요청할 수 있는
PC/모바일 실시간 미리보기 기반 소상공인 셀프 랜딩페이지 제작기.
```

MVP 개선 우선순위:

1. 기존 MVP에 업종 선택 단계를 추가한다.
2. 양식 선택을 업종별 추천 양식 그리드로 바꾼다.
3. PC 미리보기와 모바일 미리보기를 동시에 보여주는 레이아웃으로 확장한다.
4. 양식별 입력 필드를 동적으로 바꾸는 구조를 만든다.
5. AI 수정 요청 입력창과 변경 이력/적용 버튼을 추가한다.
6. 구매 전 워터마크, 구매 후 링크 발급 상태를 UI로 표현한다.

## AI 토큰 비용 최소화 원칙

사용자는 이 사업의 핵심 수익성 조건을 `최대한 선택지를 선택하는 방식으로 만들어 AI 토큰 소모를 줄이는 것`으로 판단했다.

핵심 원칙:

- AI를 매번 자유 생성에 쓰면 원가가 올라가고 결과 일관성도 낮아진다.
- 사용자는 가능한 한 버튼/카드/드롭다운/토글을 선택한다.
- AI는 꼭 필요한 순간에만 짧게 호출한다.
- 대부분의 결과물은 `업종별 스키마 + 양식 데이터 + 선택 옵션 + 규칙 기반 렌더링`으로 만든다.
- 자연어 AI 수정 요청은 유료 전환 직전 또는 제한된 횟수로 제공한다.

권장 UX:

1. 업종 선택: 카드 선택.
2. 양식 선택: 업종별 3~5개 카드 선택.
3. 분위기 선택: 고급스러운, 귀여운, 깔끔한, 감성적인, 강한 이벤트형 등 프리셋 선택.
4. 목적 선택: 예약 문의, 가격표 안내, 이벤트 홍보, 후기/사례 강조 등 선택.
5. 섹션 순서 선택: 추천 순서 기본값 + 드래그/토글.
6. 문구 선택: AI 생성보다 미리 준비된 문구 후보 3~5개 중 선택.
7. AI 수정 요청: 마지막 단계에서만 “더 고급스럽게”, “이벤트 강조” 같은 제한된 요청 제공.

토큰 비용 절감 구조:

- 1차 생성은 AI가 아니라 템플릿 엔진이 처리한다.
- 문구는 업종/양식/목적별 후보 문장 라이브러리를 미리 만들어 둔다.
- AI 호출은 `선택값 요약 + 기존 문구 + 수정 요청`처럼 작은 컨텍스트로 제한한다.
- 이미지 배치, 섹션 구성, CTA 위치는 AI가 아니라 사전 정의 규칙으로 처리한다.
- 사용자는 무료/유료 구독 구조보다 `단건 구매` 방식으로 팔 계획이다.
- 따라서 AI 사용량 제한은 무료/유료 티어 기준이 아니라 `단건 상품에 포함된 수정 범위` 기준으로 설계한다.

사업성 관점:

```text
AI가 모든 것을 매번 새로 만드는 서비스가 아니라,
잘 만든 업종별 선택지와 템플릿 데이터베이스를 기반으로
AI는 마지막 보정과 개인화에만 쓰는 서비스여야 마진이 남는다.
```

MVP 설계 반영:

1. AI 채팅창보다 선택형 옵션 UI를 먼저 만든다.
2. 업종별 프리셋 데이터와 문구 후보를 자산으로 축적한다.
3. AI 수정 요청은 “제안 버튼” 중심으로 시작한다: 더 고급스럽게, 이벤트 강조, 가격표 위로, 문구 짧게, 예약 버튼 강조.
4. 자유 입력형 AI 수정은 고급 옵션으로 둔다.
5. 토큰을 쓰는 모든 기능은 사용자가 명확히 가치 있다고 느끼는 지점에만 배치한다.

## 단건 판매 플로우와 수정 과금 정책

사용자는 구독형 무료/유료 정책보다 `단건 판매` 모델을 우선한다. 확정된 판매 흐름은 다음과 같다.

```text
1. 업종 선택
2. 양식 선택
3. 정보 입력
4. 페이지 구매 / 워터마크 제거
5. 수정 과금
```

기본 페이지 구매 가격 후보:

```text
약 30,000원
```

기본 구매에 포함되는 것:

- 선택한 업종/양식 기반 모바일 랜딩페이지 1개.
- 사용자가 입력한 사진, 가격 정보, 이벤트 정보, 랜딩 문구 반영.
- PC/모바일 실시간 미리보기 후 구매.
- 구매 시 워터마크 제거.
- 공유 링크 발급.

수정 과금 정책:

1. 디테일 수정
   - 가격: 1회당 3,000원.
   - AI 활용 시 적용.
   - 예: “좀 더 고급스럽게”, “이벤트를 더 강조”, “20대 여성 취향으로”, “가격표를 위로”, “문구를 더 짧게” 같은 디자인/카피/배치 방향 수정.

2. 정보 수정
   - 가격: 1항목당 1,000원.
   - AI 없이 데이터 치환/렌더링으로 처리 가능한 수정.
   - 예: 가격 정보 변경, 이벤트 정보 변경, 랜딩페이지 문구 일부 변경, 영업시간/연락처/예약 링크 변경.

사업성 관점:

- 최초 구매가 약 3만원이면 네이버 파워링크 유입에서도 너무 낮은 단가가 아니고, 단건 상품으로 이해시키기 쉽다.
- 이후 수정 과금을 분리하면 “한 번 팔고 끝”이 아니라 이벤트/가격표 변경 때마다 소액 매출이 발생한다.
- AI가 필요한 디테일 수정과 단순 정보 수정의 가격을 분리하는 것이 중요하다.
- 단순 정보 수정은 토큰 비용 없이 처리하므로 마진이 높다.
- 디테일 수정은 AI 토큰과 결과 검수 비용이 있으므로 3,000원 이상으로 분리 과금한다.

UX 반영:

- 구매 전에는 워터마크 미리보기.
- 구매 후에는 “정보 수정”과 “디테일 수정” 버튼을 분리해서 보여준다.
- 정보 수정은 항목별 선택형 폼으로 처리한다.
- 디테일 수정은 추천 버튼 + 제한된 자연어 요청으로 처리한다.
- 수정 비용은 요청 전에 명확히 보여준다.

## 양식 대량 생산과 현실적인 공유/배포 방식

사용자는 이 사업을 성립시키려면 `양식 이미지와 양식 페이지를 많이, 아주 많이 뽑아야 한다`고 판단했다. 업종별 양식 썸네일/미리보기 이미지와 실제 렌더링 가능한 양식 페이지가 충분히 쌓여야 사용자가 선택할 이유가 생긴다.

현실적인 구현 판단:

- 기술적으로는 충분히 가능하다.
- 다만 고객에게 `HTML 파일` 자체를 주는 방식은 일반 소상공인에게 적합하지 않다.
- 고객이 원하는 것은 파일이 아니라 `인스타 프로필/문자/카톡에 붙일 수 있는 링크`다.
- 따라서 기본 제공물은 HTML 다운로드가 아니라 `호스팅된 공유 링크`여야 한다.
- HTML 다운로드는 백업/고급 옵션 또는 크몽 납품용 보조 수단으로만 둔다.

권장 배포 구조:

```text
사용자 입력 데이터 + 선택 양식
→ 서버에서 정적 HTML/JSON 렌더링
→ 고유 URL 발급
→ 구매 시 워터마크 제거
→ 고객은 링크를 인스타 프로필/블로그/카톡/문자에 공유
```

예상 링크 형태:

```text
https://서비스도메인.com/p/abc123
https://서비스도메인.com/nail/on-nail-seongsu
```

구현 방식 후보:

1. 정적 페이지 생성 방식
   - 구매 시 HTML 파일을 생성해서 서버/S3/Cloudflare Pages/R2 등에 저장한다.
   - 빠르고 저렴하며 단건 판매와 잘 맞는다.

2. DB 렌더링 방식
   - URL 접속 시 DB의 사용자 데이터와 양식 ID를 불러와 렌더링한다.
   - 수정 반영이 쉽고 관리 화면을 만들기 좋다.

3. 혼합 방식
   - 편집/수정 중에는 DB 렌더링.
   - 구매/확정 후에는 정적 HTML로 발행.
   - 초기 MVP에는 이 방식이 가장 현실적이다.

판매/UX 원칙:

- 사용자에게는 “HTML 파일”이라고 설명하지 말고 “공유 링크 발급”이라고 설명한다.
- “워터마크 제거”는 링크 발급/공개 상태와 연결한다.
- 링크 발급 후에도 정보 수정은 1항목 1,000원, 디테일 수정은 1회 3,000원으로 과금한다.
- 양식 썸네일은 대량 생산해야 하며, 실제 페이지와 1:1로 연결되어야 한다.

다음 제작 방향:

1. 업종별 양식 썸네일/미리보기 이미지 대량 생성 체계 만들기.
2. 업종별 실제 렌더링 가능한 HTML 양식 페이지 대량 생성.
3. 사용자 데이터 JSON과 양식 ID를 조합해 고유 URL을 발급하는 구조 설계.
4. HTML 다운로드보다 공유 링크 발급을 기본 납품 방식으로 바꾸기.

## 도메인/링크 발급 비용 정책

사용자는 링크 발급 방식이면 도메인 유지비가 발생한다는 점을 우려했다.

정리:

- 고객마다 별도 도메인을 사주는 방식은 기본 상품에 넣으면 안 된다.
- 기본 상품은 서비스가 보유한 대표 도메인 1개 아래의 경로 또는 서브도메인으로 링크를 발급한다.
- 도메인 유지비는 고객별 비용이 아니라 서비스 공통 고정비로 처리한다.
- 고객이 자기 도메인을 원할 때만 별도 유료 옵션으로 제공한다.

기본 링크 구조 후보:

```text
https://서비스도메인.com/p/abc123
https://서비스도메인.com/nail/on-nail-seongsu
https://go.서비스도메인.com/abc123
```

비용 구조:

- 대표 도메인 1개: 연 단위 고정비.
- 고객별 페이지: 정적 파일/DB 레코드/이미지 저장 비용만 발생.
- 기본 3만원 상품에 고객별 도메인 비용을 포함하지 않는다.
- 커스텀 도메인 연결은 추가 옵션으로 분리한다.

커스텀 도메인 옵션 예시:

```text
기본: 서비스 도메인 링크 발급 포함
추가: 개인 도메인 연결/설정 대행 별도 과금
```

권장 판매 표현:

- “공유 링크 발급”이라고 말한다.
- “개인 도메인 제공”이라고 말하지 않는다.
- “인스타 프로필에 넣을 수 있는 링크”를 기본 가치로 강조한다.
- 커스텀 도메인은 필요한 고객에게만 상위 옵션으로 안내한다.

## 단일 도메인 페이지 수용량 판단

사용자는 도메인 1개 아래 고객별 페이지를 계속 늘리는 방식에서 `도메인 하나에 페이지를 몇 개까지 연결 가능한지`를 확인했다.

정리:

- 도메인 자체에는 페이지 개수 제한이 사실상 없다.
- `서비스도메인.com/p/abc123`처럼 경로 기반 URL을 쓰면 고객 페이지를 수천~수백만 개까지도 논리적으로 만들 수 있다.
- 실제 한계는 도메인이 아니라 서버/호스팅/DB/스토리지/트래픽/관리 구조다.
- 정적 HTML + CDN 구조를 쓰면 초기 단건 판매 서비스에는 페이지 수 제한이 거의 문제 되지 않는다.

권장 URL 구조:

```text
https://서비스도메인.com/p/abc123
https://서비스도메인.com/p/on-nail-seongsu
https://서비스도메인.com/nail/on-nail-seongsu
```

주의할 점:

- 서브도메인 방식 `customer.서비스도메인.com`은 DNS/SSL/관리 복잡도가 올라갈 수 있으므로 초기에는 경로 방식 `/p/고객ID`가 더 적합하다.
- 페이지가 많아질 때 중요한 것은 URL 충돌 방지, 비공개/공개 상태 관리, 삭제/만료 정책, 이미지 저장 비용, 트래픽 비용이다.
- 초기 상품은 “서비스 도메인 아래 공유 링크 발급”으로 안내하고, 고객별 개별 도메인/서브도메인은 기본 상품에 포함하지 않는다.

## 랜딩페이지 데이터 수집/전환 측정의 중요성

사용자는 랜딩페이지의 핵심 가치가 `클릭률, 전환률 같은 데이터 수집`에 있다고 판단했다.

정리:

- 맞다. 단순히 예쁜 모바일 페이지를 제공하는 것만으로는 장기 가치가 약하다.
- 랜딩페이지의 본질은 방문자가 보고, 클릭하고, 문의/예약으로 이어지는 흐름을 측정하고 개선하는 것이다.
- 다만 초기 소상공인용 단건 상품에서는 너무 복잡한 마케팅 대시보드보다 `몇 명이 봤고, 몇 명이 눌렀는지`를 쉽게 보여주는 정도가 적합하다.

초기 MVP에서 수집해야 할 최소 데이터:

1. 페이지 조회수: 페이지가 몇 번 열렸는지.
2. 고유 방문자 수: 중복을 줄인 대략적인 방문자 수.
3. CTA 클릭 수: 카톡 문의, 전화, 예약, 지도, 인스타 버튼 클릭 수.
4. CTA 클릭률: 방문자 대비 버튼 클릭 비율.
5. 유입 경로: 인스타, 네이버, 카톡, 문자, 직접 접속 등 가능한 범위.
6. 기기 구분: 모바일/PC 비율.

소상공인에게 보여줄 표현:

```text
이번 달 페이지 조회 320회
카톡 문의 버튼 클릭 28회
전화 버튼 클릭 7회
예약 전환 클릭률 10.9%
```

사업적 의미:

- 데이터 제공이 들어가면 단순 “페이지 판매”보다 가치가 올라간다.
- 고객은 수정 과금의 이유를 이해하기 쉬워진다. 예: 이벤트 버튼 클릭률이 낮으니 디테일 수정 3,000원으로 강조도를 높인다.
- 업종별/양식별 평균 클릭률 데이터를 쌓으면 어떤 양식이 잘 팔리고 잘 전환되는지 알 수 있다.
- 장기적으로는 `전환 잘 되는 업종별 양식 추천`이 핵심 경쟁력이 된다.

주의점:

- 초반부터 과도한 애널리틱스/광고 대시보드를 만들 필요는 없다.
- 개인정보/쿠키 동의 이슈를 피하려면 초기에는 개인 식별 없이 페이지 조회와 버튼 클릭 이벤트 중심으로 간단히 수집한다.
- 고객에게는 복잡한 지표보다 “조회수, 클릭수, 문의 버튼 클릭률”만 보여주는 것이 좋다.

제품 방향 업데이트:

```text
소상공인 모바일 랜딩페이지 제작기 + 간단한 클릭/전환 데이터 리포트.
```

이 기능이 붙으면 단건 판매 후에도 고객이 다시 들어와 확인하고, 정보 수정/디테일 수정으로 이어질 가능성이 커진다.

## 데이터 수집을 위한 DB 필요성

사용자는 클릭률/전환률 데이터를 남기려면 DB 연결이 필요한지 확인했다.

정리:

- 고객 페이지 조회수, CTA 클릭수, 수정 이력, 구매 상태, 워터마크 제거 여부를 관리하려면 결국 DB가 필요하다.
- 하지만 처음부터 복잡한 백엔드/대형 DB를 만들 필요는 없다.
- MVP는 `페이지 데이터 + 이벤트 로그`만 저장하는 단순 DB로 충분하다.
- 정적 HTML만 배포해도 이벤트 수집 API 또는 서버리스 엔드포인트로 클릭/조회 데이터를 저장할 수 있다.

초기 DB에 필요한 최소 테이블/컬렉션:

1. pages
   - page_id
   - industry
   - template_id
   - customer_name
   - input_data JSON
   - is_paid
   - watermark_removed
   - public_url
   - created_at
   - updated_at

2. events
   - event_id
   - page_id
   - event_type: page_view, cta_click, phone_click, kakao_click, map_click 등
   - button_id
   - referrer
   - device_type
   - created_at

3. orders 또는 payments
   - order_id
   - page_id
   - amount
   - status
   - purchased_at

4. revisions
   - revision_id
   - page_id
   - revision_type: info_update 또는 ai_detail_update
   - amount
   - change_summary
   - created_at

권장 MVP 구조:

```text
프론트 편집기
→ pages DB에 사용자 입력/양식 저장
→ 미리보기 렌더링
→ 구매 시 is_paid=true, watermark_removed=true
→ 발급된 페이지에 조회/클릭 추적 스크립트 삽입
→ events DB에 page_view/cta_click 저장
→ 사장님용 간단 리포트에서 집계 표시
```

저비용 구현 후보:

- Supabase: Postgres + 인증/스토리지/API를 빠르게 붙이기 좋다.
- Firebase: 이벤트성 데이터/간단 앱에 빠르다.
- Cloudflare D1 + R2 + Pages/Workers: 정적 페이지 발행과 저비용 운영에 적합하다.
- SQLite + 단일 서버: 아주 초기 내부 MVP에 적합하다.

제품 원칙:

- 고객에게 DB/기술을 설명하지 않는다.
- 고객에게는 “조회수와 문의 클릭을 확인할 수 있습니다”라고 표현한다.
- 개인정보를 깊게 수집하지 말고, 초기에는 개인 식별 없는 조회/클릭 이벤트 중심으로 간단히 시작한다.

## 랜딩페이지 사용 기한 정책

사용자는 구매한 랜딩페이지에 사용 기한을 두는 방식을 검토했다. 예: 1개 구매 시 5년 사용 또는 10년 사용.

판단:

- 사용 기한을 두는 것은 좋다. 링크 호스팅/데이터 저장/분석 제공에는 장기 운영비와 관리 책임이 있으므로 “평생 제공”은 피해야 한다.
- 다만 3만원 단건 상품에 5년/10년을 기본 제공하면 장기 유지 책임이 너무 커질 수 있다.
- 소상공인 입장에서는 1년은 짧게 느껴질 수 있고, 5년은 충분히 길게 느껴진다.
- 현실적인 기본안은 `기본 3년 제공`, 상위 옵션으로 `5년 제공`, 별도 장기 옵션으로 `10년 제공`을 두는 것이다.

권장 상품 구조:

```text
기본 페이지 구매: 약 30,000원 / 링크 3년 유지
장기 유지 옵션: 5년 사용권 추가 과금
프리미엄 장기 옵션: 10년 사용권 추가 과금
```

대안:

```text
30,000원: 3년 사용
39,000원~49,000원: 5년 사용
59,000원~79,000원: 10년 사용
```

판매 문구:

```text
구매 후 3년 동안 공유 링크가 유지됩니다.
장기 사용이 필요한 경우 5년/10년 유지 옵션을 선택할 수 있습니다.
```

주의점:

- “영구 사용”, “평생 유지” 표현은 피한다.
- 사용 기한 종료 전 연장 결제 안내를 할 수 있어야 한다.
- 링크 만료 후에도 데이터 백업/재발행 정책을 정해야 한다.
- 단건 판매 가격에는 호스팅/스토리지/트래픽/관리 비용이 포함되어 있음을 내부 원가 계산에 반영해야 한다.
